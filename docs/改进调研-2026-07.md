# Grok Coder 改进调研（2026-07-09）

基线：v0.3.1 + 未提交的品牌清理改动。当前状态健康——649 个测试全绿、tsc 干净、上游 phuryn/grok-build-vscode 仍停在 1.4.30（无新提交需要同步）。近几轮优化质量不错：CLI 登录态复用（`grok-cli-auth.ts`，纯函数 + 测试）、启动时跳过失效默认模型（`acp.ts` + 测试）、品牌清理均已落地。以下按优先级列出改进点。

## P0 — 正确性风险（建议尽快修）

**1. Cursor 兼容守卫（最重要）。**
（范围决策：旧版 VS Code 兼容**不做**，`engines: ^1.104` 保持不变、不引入类型 shim；但 Cursor 必须支持。）
问题不在版本，而在 API 存在性：`vscode.lm.registerLanguageModelChatProvider` / `vscode.chat.createChatParticipant` 由 Copilot Chat 体系提供，**Cursor 里根本没有**——`activateDirect` 无守卫地调用它们，activate 直接抛异常，**连上游的侧边栏也一起挂掉**（上游本来支持 Cursor，README 明确宣传）。修法：

- `activateDirect` 内按功能分块 try/catch + 能力探测：`typeof (vscode as any).lm?.registerLanguageModelChatProvider === 'function'` 才注册 provider，`typeof (vscode as any).chat?.createChatParticipant === 'function'` 才注册 participant；SCM 提交按钮和状态栏不依赖这些 API，在 Cursor 里应照常工作（direct 直连 + CLI 登录复用是 Cursor 用户的主要价值点）；
- 任何一块注册失败只禁用该块并写日志，绝不影响侧边栏；
- 验证项：在真实 Cursor 上装 vsix，确认侧边栏 + SCM 按钮 + 状态栏可用、provider/participant 静默缺席。若 Cursor 的 VS Code 基线低于 1.104 导致装不上，再回头讨论 engines（当前不做预案）。

**2. OAuth 刷新无并发保护。**
`getValidAccessToken` 过期时发起刷新，但 provider 和 participant 可能同时触发——两个并发 refresh 用同一个 refresh_token，xAI 若做 refresh-token 轮换会导致其中一个失败甚至凭据失效。修法：模块级 in-flight promise 去重（同 upstream `primingPromise` 的手法）。

**3. CLI token 与 API 权限面未经实测验证。**
`grokCli` 模式把 `~/.grok/auth.json` 的 access token 直接用于 `/v1/responses`。client_id 相同，理论可行，但 CLI token 的 scope（`grok-cli:access`）对直连 API 是否放行、部分订阅档位是否 403，都需要真机验证。建议：在 403 时自动降级重试下一档认证（grokCli → oauth → apiKey），而不是直接把错误抛给用户（现在 `getAuthToken` 选中 grokCli 后就不再尝试后两档了——只有 auth.json 里 token 缺失/过期才会走到下一档）。

**4. `reasoning.effort` 对非推理模型会 400。**
`additionalModels` 里若配了不支持 reasoning 的模型（如 grok-code-fast 系），请求仍带 `reasoning: { effort }`，xAI 会拒。参照 pi-grok 的 `supportsReasoningEffort` 做模型白名单/配置项开关。

## P1 — 代码质量

**5. direct 模块测试覆盖不足。**
目前只有 `grok-cli-auth` 有测试。上游的核心哲学就是"纯函数拆出来、协议行为不起进程就能测"，direct 里现成的纯函数都值得补：`xaiClient` 的 SSE 行解析与 `sanitizeInput`、`provider` 的 `convertMessages/convertTools`（含 system→instructions、图像、工具调用回放）、`oauth` 的 `validateEndpoint/toCredentials`、`commit` 的 `stripFences`。这也是防上游合并回归的安全网。建议把 SSE 解析从 `XaiClient.stream` 里拆成可独立测试的纯函数。

**6. direct 模块用户可见文案未走 l10n。**
所有提示/QuickPick/participant 回复都是硬编码中文——英文用户体验差，且与 package.nls 双语化不一致。改用 `vscode.l10n.t()` + `l10n/bundle.l10n.zh-cn.json`（package.json 加 `"l10n": "./l10n"`），默认英文、中文翻译。participant 的系统提示词可保留"回答使用与用户相同的语言"策略。

**7. commit.ts 多仓库工作区取 `repositories[0]`。**
多 repo workspace 下可能生成错仓库的提交信息。按活动编辑器归属仓库选择，多个则 QuickPick。另外 `getGitRepo` 返回 `any`，可引入 git 扩展的类型声明（upstream 惯例是最小依赖，复制官方 `git.d.ts` 即可）。

**8. fork 历史与上游断链。**
仓库历史被重置（仅 2 个提交），`git merge upstream/main` 无共同祖先。建议在 CLAUDE.md 记录同步流程：`git remote add upstream …` + `git diff v1.4.30..upstream/main -- src/ media/ | git apply` 的 patch 式同步（我们改动集中在 `src/direct/`，冲突面小），并在每次同步后重跑 649 测试底线。

**9. 小项。** `auth.ts` 每次请求都同步读 auth.json（可按 mtime 缓存几秒）；`xaiClient` 流无 stall 超时（网络挂起会一直等）；`statusbar` 的 `grokCoder.statusBarMenu` 命令未在 package.json 声明（命令面板搜不到，功能正常但不规范）。

## P2 — 功能机会（按性价比排序）

**10. Prompt caching：一行配置省 75% 输入成本。**
xAI 支持 `prompt_cache_key`（缓存输入 $0.5/M vs $2/M）。原生 Chat 的多轮对话每轮都重发全量历史——给每个 VS Code 会话生成稳定的 cache key 传入即可，收益直接。

**11. 状态栏升级为"活值"。**
现在显示的是配置默认值。上游 roadmap 第 2 项想要的是当前会话的实况：live model、effort、token 用量（sidebar 已有 `_meta.totalTokens` 数据流）。让 `GrokSidebar` 暴露一个 EventEmitter，状态栏订阅即可，改动面小、是真差异化。

**12. Web/X 搜索扩展到默认对话。**
目前只有 `/search` 用服务端搜索工具。可加配置 `grokCoder.autoSearch`：默认对话也带 `web_search`/`x_search`（xAI 独有卖点），由模型自行决定是否搜。注意在 README 标注工具计费。

**13. `@grok /fix` 结果一键应用。**
participant 返回的修复代码目前只能手动复制。用 `stream.button` + 命令把代码块 diff 应用到编辑器（`WorkspaceEdit`），闭环体验明显提升。

**14. Webview 中文化（既定 v0.4 目标）。**
`media/chat.js` 全部英文硬编码。方案：host 把 locale 传进 webview，chat.js 加一层 `t()` 字典。工作量大但对中文市场是核心卖点；建议做成字典文件以便跟上游 merge 时冲突最小。

**15. 观望项。** xAI 新 API（context compaction、WebSocket mode、priority processing）暂无明确插件场景，不建议现在接；inline completions（grok-code-fast 直连）值得排在 webview 中文化之后评估。
